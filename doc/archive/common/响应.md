# 错误码

## 组成

- [ ] 长度定义：X 位
- [ ] 责任方标识：0-无责任方（且错误码固定为0...0）、1-调用方服务或用户、2-业务方服务、3-第三方服务
- [ ] 宏观错误码标识-微服务：00
- [ ] 宏观错误码标识-模块功能：00
- [ ] 流水标识：Feedback：00000

## 分类

- [ ] 0...0：成功
- [ ] 认证授权 A
- [ ] 业务错误 B
- [ ] 缓存错误 C
- [ ] 数据库错误 D
- [ ] 文件IO错误 F
- [ ] 网络错误 N
- [ ] 其他错误 O
- [ ] 参数错误 P
- [ ] RPC错误 R_(0:请求|1:返回)_业务_方法_调用方CODE码
- [ ] 系统错误 S
- [ ] 9...9：极端错误

## 管理

- [ ] 很多大型公司，会有统一开发平台，上面可以维护错误码，这样所以服务组件可以快捷查询、新建错误码，也不会重现重复的情况，当然，这一般都是中大型公司会有。
- [ ] 开发者申请[宏观错误码标识-微服务]
- [ ] 开发者申请[宏观错误码标识-模块功能]
- [ ] 开发者提供 责任方标识、流水标识、错误描述 上报至 [错误码管理系统]，并获得完整错误码

## 参考

Alibaba开发手册：

- 【强制】错误码为字符串类型，共5位，分成两个部分：错误产生来源+四位数字编号。
  说明：错误产生来源分为 A/B/C，A表示错误来源于用户，比如参数错误，用户安装版本过低，用户支付超时等问题；
  B 表示错误来源于当前系统，往往是业务逻辑出错，或程序健壮性差等问题；C表示错误来源于第三方服务，比如CDN
  服务出错，消息投递超时等问题；四位数字编号从0001到9999，大类之间的步长间距预留100，参考哀来附表3支术，
- 5.【强制】编号不与公司业务架构，更不与组织架构挂钩，以先到先得的原则在统一平台上进行，审批生
  效，编号即被永久固定。
- 【参考】对于公司外的 http / api 开放接口必须使用错误码，而应用内部推荐异常抛出；跨应用间
  RPC调用优先考虑使用 Result 方式，封装isSuccess（方法、错误码、错误简短信息；应用内部推荐
  异常抛出。
  说明：关于 RPC 方法返回方式使用 Result 方式的理由：
  1）使用抛异常返回方式，调用方如果没有捕获到就会产生运行时错误。
  2） 如果不加栈信息，只是 new 自定义异常，加入自己的理解的 errorDetail message，对于调用端解决问题的帮助不会太多。
  如果加了栈信息，在频繁调用出错的情况下，数据序列化和传输的性能损耗也是问题。
