# 依赖版本管理设计思路

## 1. 总体架构

采用 **BOM + Parent 双轨制** 实现依赖版本与构建配置的解耦管理。

```
bubbles-engine (聚合根)
└── bubbles-dependency (依赖管理聚合)
    ├── bubbles-bom    (纯依赖版本声明)
    └── bubbles-parent (继承 + 构建配置)
```

## 2. 模块职责

### 2.1 bubbles-bom
- **定位**：纯 BOM（Bill of Materials），`packaging=pom`。
- **职责**：在 `dependencyManagement` 中集中声明所有第三方依赖及版本号。
- **特点**：无 parent 继承，可被外部项目直接 `import`，实现版本统一管理。

### 2.2 bubbles-parent
- **定位**：业务模块的父 POM。
- **职责**：
  1. **继承外部依赖**：继承 `dante-dependencies`，获取基础依赖版本。
  2. **覆盖内部版本**：通过 `import bubbles-bom`，利用 Maven `import` 优先级高于 `parent` 的机制，覆盖外部依赖中需要调整的版本。
  3. **统一构建配置**：定义 JDK 版本、编译参数、插件版本、代码格式化（Spotless）、环境 Profile 等。

## 3. 版本覆盖机制

Maven 的 `dependencyManagement` 优先级规则：

```
当前 POM 直接声明 > import 引入 > parent 继承
```

因此，在 `bubbles-parent` 中 `import bubbles-bom` 后，BOM 内声明的依赖版本将**覆盖** `dante-dependencies` 中的同名依赖版本，实现精准管控。

## 4. 使用方式

业务模块仅需继承 `bubbles-parent`，无需显式声明版本：

```xml
<parent>
    <groupId>com.bubbles.engine</groupId>
    <artifactId>bubbles-parent</artifactId>
    <version>${project.version}</version>
</parent>

<dependencies>
    <dependency>
        <groupId>xxx</groupId>
        <artifactId>xxx</artifactId>
    </dependency>
</dependencies>
```

## 5. 核心收益

| 收益 | 说明 |
|------|------|
| **版本集中** | 所有第三方依赖版本收敛于 `bubbles-bom` 一处 |
| **灵活覆盖** | 通过 `import` 机制安全覆盖外部 parent 的版本 |
| **外部可复用** | `bubbles-bom` 可被非 bubbles 体系的项目直接引用 |
| **构建统一** | 编译、格式化、插件等行为全局一致 |
